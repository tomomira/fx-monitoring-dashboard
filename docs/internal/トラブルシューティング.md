## 1. PowerShell改行エラー（バックスラッシュ問題）

### 発生した問題

PowerShellでBash形式のコマンド（バックスラッシュ `\` で改行）を実行すると、以下のエラーが発生：

```
run.googleapis.com : 用語 'run.googleapis.com' は、コマンドレット、関数、スクリプト ファイル、または操作可能なプログラ
ムの名前として認識されません。名前が正しく記述されていることを確認し、パスが含まれている場合はそのパスが正しいことを確
認してから、再試行してください。
```

**発生例**:
```powershell
# エラーになるコマンド
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  cloudbuild.googleapis.com \
  cloudscheduler.googleapis.com
```

### 原因

- PowerShellでは改行にバックスラッシュ `\` が使えない
- Bash/Zsh用の構文がPowerShellでは動作しない
- ドキュメントがLinux/Mac向けに書かれていた

### 解決方法

#### 方法1: バックティックを使用（PowerShell用）

```powershell
gcloud services enable `
  run.googleapis.com `
  artifactregistry.googleapis.com `
  cloudbuild.googleapis.com `
  cloudscheduler.googleapis.com
```

#### 方法2: 1行で記述（推奨・確実）

```powershell
gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com cloudscheduler.googleapis.com
```

### 教訓

- **Windows環境ではPowerShell用のコマンド例を記載する**
- **QUICKSTART.mdをPowerShell対応に全面修正**
- **Linux/Mac用はdetailsタグで折りたたんで補足**

## 2. venvフォルダアップロードエラー

### 発生した問題

Cloud Run Jobsへのデプロイ時に以下のエラーが発生：

```
ERROR: gcloud crashed (OSError): [WinError 1920] ファイルにアクセスできません。: '.\\venv\\bin\\python'
```

### 原因

- `venv`フォルダ（Python仮想環境）がCloud Buildにアップロードされた
- WindowsのPython実行ファイル（`venv/Scripts/python.exe`）がLinuxコンテナで実行できない
- `.gcloudignore` ファイルが存在しなかった

### 解決方法

`.gcloudignore` ファイルを作成し、venvフォルダを除外：

```gitignore
# Python仮想環境を除外
venv/
__pycache__/
*.pyc
*.pyo
*.pyd
.Python

# エディタ・IDE設定ファイル
.vscode/
.idea/
*.swp
*.swo
*~

# OS生成ファイル
.DS_Store
Thumbs.db

# Git関連
.git/
.gitignore

# ドキュメント（デプロイ不要）
README.md
QUICKSTART.md

# その他
*.log
.env
```

**再デプロイ**:
```powershell
gcloud run jobs deploy fx-mxn-watcher-slack --source . --region asia-northeast1
```

### 教訓

- **`.gcloudignore` はプロジェクト作成時に必ず作成する**
- **venv, __pycache__, .vscode などは必ず除外する**
- **ドキュメントファイルも除外してデプロイを高速化**

## 3. 環境変数が反映されない問題

### 発生した問題

環境変数を設定したはずなのに、ログに以下のメッセージが表示される：

```
[!] SLACK_WEBHOOK_URL is not set. Skipping Slack notification.
```

環境変数確認コマンドでも何も表示されない：
```powershell
gcloud run jobs describe fx-mxn-watcher-slack --region=asia-northeast1 --format="value(template.template.containers[0].env)"
# 何も表示されない
```

### 原因

- `gcloud run jobs deploy` コマンドが環境変数をクリアする仕様
- デプロイ前に設定した環境変数は、デプロイ後に消える
- `--set-env-vars` でカンマ区切りで設定すると、正しく解析されない場合がある

### 解決方法

#### Step 1: 環境変数をクリア

```powershell
gcloud run jobs update fx-mxn-watcher-slack --clear-env-vars --region asia-northeast1
```

#### Step 2: 個別に設定（推奨）

```powershell
# OpenAI APIキーを設定
gcloud run jobs update fx-mxn-watcher-slack --update-env-vars OPENAI_API_KEY="sk-..." --region asia-northeast1

# Slack Webhook URLを設定
gcloud run jobs update fx-mxn-watcher-slack --update-env-vars SLACK_WEBHOOK_URL="https://hooks.slack.com/services/..." --region asia-northeast1
```

#### Step 3: 設定を確認

```powershell
gcloud run jobs describe fx-mxn-watcher-slack --region=asia-northeast1
```

`Env vars:` セクションに以下が表示されることを確認：
```
Env vars:
  OPENAI_API_KEY   sk-proj-...
  SLACK_WEBHOOK_URL https://hooks.slack.com/services/...
```

### 教訓

- **デプロイ後に必ず環境変数を設定する**
- **`--set-env-vars` より `--update-env-vars` を個別に実行する方が確実**
- **設定後は必ず `describe` コマンドで確認する**

## 4. Slack通知が届かない問題

### 発生した問題

ログには以下のメッセージが表示されるが、Slackに通知が届かない：

```
[✓] Slack notification sent successfully.
```

### 原因と確認方法

#### 原因1: Webhook URLが無効

- Webhook URLの有効期限が切れている
- Incoming Webhooksアプリが削除された
- Webhook URLのタイプミス

**確認方法**:
```powershell
# PowerShellで直接テスト送信
$body = '{"text":"テスト通知：AI FX Watcher Bot"}'
Invoke-WebRequest -Uri "https://hooks.slack.com/services/..." -Method Post -Body $body -ContentType "application/json"
```

#### 原因2: 通知チャンネルが間違っている

- Webhook URL作成時に指定したチャンネルを確認
- 複数のワークスペースを使用している場合、別のワークスペースに送信されている可能性

**確認方法**:
1. Slack > ワークスペース名 > 「設定と管理」 > 「アプリを管理」
2. 「Incoming Webhooks」を検索
3. 設定を開いて送信先チャンネルを確認

#### 原因3: Slackの検索で見つからない

- Slackアプリの「すべての未読」で確認
- Slackの検索バーで `AI FX Watcher` で検索

### 解決方法

#### 新しいWebhook URLを発行

1. Slack > Incoming Webhooks アプリ
2. 「新しいWebhookを追加」
3. 通知先チャンネルを指定
4. 表示されたWebhook URLをコピー
5. 環境変数を更新：

```powershell
gcloud run jobs update fx-mxn-watcher-slack --update-env-vars SLACK_WEBHOOK_URL="https://hooks.slack.com/services/新しいURL" --region asia-northeast1
```

### 教訓

- **Webhook URLは定期的に動作確認する**
- **テスト送信コマンドでWebhook URLの有効性を確認**
- **通知チャンネルを明確に記録しておく**

## 5. Scheduler経由で環境変数が反映されない

### 発生した問題

- `gcloud run jobs execute` （直接実行）: ✅ 成功、Slack通知が届く
- `gcloud scheduler jobs run` （Scheduler経由）: ❌ 失敗、`SLACK_WEBHOOK_URL is not set`

### 原因

- Schedulerが古いCloud Run Jobのリビジョンを参照している
- 環境変数を更新しても、Schedulerのキャッシュに古い設定が残っている
- Schedulerの内部的なキャッシュの問題

### 解決方法

#### 方法1: 時間が経つのを待つ（推奨）

- 通常、数時間〜1日で自動的に新しいリビジョンが使われるようになる
- 次の毎時0分の自動実行で確認

#### 方法2: Schedulerを再作成（確実）

```powershell
# 1. 既存のSchedulerを削除
gcloud scheduler jobs delete fx-mxn-watcher-slack-schedule --location=asia-northeast1

# 2. プロジェクトIDを取得
$PROJECT_ID = gcloud config get-value project

# 3. Schedulerを再作成
gcloud scheduler jobs create http fx-mxn-watcher-slack-schedule --location=asia-northeast1 --schedule="0 * * * *" --time-zone="Asia/Tokyo" --uri="https://asia-northeast1-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/${PROJECT_ID}/jobs/fx-mxn-watcher-slack:run" --http-method=POST --oauth-service-account-email="cloud-scheduler-runner@${PROJECT_ID}.iam.gserviceaccount.com"
```

### 確認方法

```powershell
# Scheduler経由で手動実行
gcloud scheduler jobs run fx-mxn-watcher-slack-schedule --location=asia-northeast1

# 30秒待ってからログ確認
gcloud logging read "resource.type=cloud_run_job AND resource.labels.job_name=fx-mxn-watcher-slack" --limit 10 --format="table(timestamp, textPayload)"
```

`[✓] Slack notification sent successfully.` が表示されればOK。

### 教訓

- **環境変数更新後は、まず直接実行（execute）でテストする**
- **Scheduler経由の動作は、次の自動実行で確認する**
- **確実性を求める場合はSchedulerを再作成する**

## 6. サービスアカウント既存エラー

### 発生した問題

サービスアカウント作成時に以下のエラーが発生：

```
ERROR: (gcloud.iam.service-accounts.create) Resource in projects [fx-mxn-watcher-bot] is the subject of a conflict: Service account cloud-scheduler-runner already exists within project projects/fx-mxn-watcher-bot.
```

### 原因

- ID084（前バージョン）で既にサービスアカウントを作成していた
- 同じプロジェクト内では同名のサービスアカウントは作成できない

### 解決方法

**このエラーは無視してOK**。既存のサービスアカウントをそのまま使用できます。

次のコマンド（権限付与）から続行：

```powershell
# プロジェクトIDを取得
$PROJECT_ID = gcloud config get-value project

# 権限付与（既存のサービスアカウントに対して実行）
gcloud projects add-iam-policy-binding $PROJECT_ID --member="serviceAccount:cloud-scheduler-runner@${PROJECT_ID}.iam.gserviceaccount.com" --role="roles/run.developer"

gcloud run jobs add-iam-policy-binding fx-mxn-watcher-slack --region=asia-northeast1 --member="serviceAccount:cloud-scheduler-runner@${PROJECT_ID}.iam.gserviceaccount.com" --role="roles/run.invoker"
```

### 教訓

- **サービスアカウントは同一プロジェクト内で再利用できる**
- **既存エラーは無視して、権限付与から続行する**
- **複数のCloud Run Jobsで同じサービスアカウントを使用してもOK**

## よくある質問 (FAQ)

### Q1. ログに `[!] SLACK_WEBHOOK_URL is not set.` と表示される

**A**: 環境変数が設定されていません。以下を実行：

```powershell
# 環境変数をクリアして再設定
gcloud run jobs update fx-mxn-watcher-slack --clear-env-vars --region=asia-northeast1
gcloud run jobs update fx-mxn-watcher-slack --update-env-vars OPENAI_API_KEY="sk-..." --region=asia-northeast1
gcloud run jobs update fx-mxn-watcher-slack --update-env-vars SLACK_WEBHOOK_URL="https://..." --region=asia-northeast1

# 直接実行してテスト
gcloud run jobs execute fx-mxn-watcher-slack --region=asia-northeast1 --wait
```

### Q2. Slack通知が届いたり届かなかったりする

**A**: Scheduler経由の実行で古いリビジョンが使われている可能性があります。Schedulerを再作成してください（問題5を参照）。

### Q3. デプロイ時に `WinError 1920` エラーが出る

**A**: `.gcloudignore` ファイルが存在しないか、venvが除外されていません。問題2を参照してください。

### Q4. PowerShellでコマンドがエラーになる

**A**: バックスラッシュ `\` をバックティック `` ` `` に変更するか、1行で記述してください（問題1を参照）。
